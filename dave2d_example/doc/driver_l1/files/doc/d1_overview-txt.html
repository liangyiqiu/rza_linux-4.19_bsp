D1 Driver Overview

The low-level driver provides a hardware and OS independent API to let D/AVE 2D applications run on any given platform.  This abstraction is needed by:

  • the generic D/AVE 2D level 2 driver
  • the TES D/AVE 2D EGL and OpenGL-ES subset implementations
  • the TES provided standard demos for D/AVE 2D
  • further D/AVE 2D application code

It is important to notice, that only the first point is mandatory: It is possible to implement a subset of the D/AVE 2D interface and implement all functions, which are not needed by the level 2 driver itself as stubs.

The functions required by the level 2 driver are

  • d1_opendevice
  • d1_closedevice
  • d1_devicesupported
  • d1_getregister
  • d1_setregister
  • d1_allocvidmem
  • d1_freevidmem
  • d1_queryarchitecture
  • d1_cacheblockflush
  • d1_mapfromvidmem
  • d1_maptovidmem
  • d1_copytovidmem
  • d1_queryirq
  • d1_allocmem
  • d1_freemem
  • d1_memsize

For a complete list of D1 types and functions see D1 API.

Memory Handling

The D/AVE 2D driver differs between normal CPU memory handled via d1_allocmem, d1_freemem and d1_memsize and ‘video memory’, handled via the ‘d1_*vidmem’ functions.  The difference between these types of memory is that CPU memory only needs to be accessible by the CPU alone and video memory needs to be accessible by both the GPU (HW) and the CPU.  Only video memory of type ‘d1_mem_dlist’ (see d1_allocvidmem) is allocated directly by the D/AVE 2D driver.

Video memory is handled differently by the D2 driver based on the return value of d1_queryarchitecture.  In general, there are three different types of memory architectures:

Unified memory systems (d1_ma_unified)

In unified memory systems, the CPU can directly access allocated video memory at the same address as D/AVE does.  Display lists are directly allocated and assembled in the video memory.  Before a display list is executed by D/AVE, the list of display list blocks is walked along and every block is flushed out of the CPU cache.

Static mapped systems (d1_ma_mapped)

Like in unified memory systems, the CPU can directly access allocated video memory.  The only difference is that the effective address is different on D/AVE side compared to CPU side.  The level 1 driver offers special mapping functions for this reason.  The display lists are handled just like on the unified memory systems.  The only difference is that the mapping functions are used to convert the addresses.

Separated memory systems (d1_ma_separated)

On systems with separated memory, the CPU can not directly access the video memory.  The level 1 driver offers special copy and dynamic mapping functions to access it.  The display list blocks are allocated and assembled in the local memory on these kinds of systems.  Before a display list block is executed by D/AVE, the whole list is walked along and every block is copied to an equally sized block in the video memory using the level 1 functions.

A special ‘low localmem’ mode is available for separated memory systems that only offer very little local memory.  On such systems it is not possible to build up the complete display list in the CPU memory.  In ‘low localmem’ mode the D/AVE driver only uses two local display list blocks.  Every time a block is completely filled, it is copied to a block in video memory asynchronously, while the second block can be immediately used by the driver again.  The final display list blocks in the video memory have a size which is a multiple of the local display list block size, so that multiple local blocks are concatenated to a single block in video memory.  Assuming that dynamic memory allocation should be avoided on such systems as much as possible, the maximum number of display list blocks in video memory is also constrained here (the driver must hold a table with the addresses of all these video memory blocks).  The ‘low localmem mode’ can be enabled and configured using the ‘d2_lowlocalmemmode’ function after a call to ‘d2_opendevice’ and before calling ‘d2_inithw’.
